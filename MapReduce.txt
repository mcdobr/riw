<h1 id="firstHeading" class="firstHeading" lang="en">MapReduce</h1>			<div id="bodyContent" class="mw-body-content">
				<div id="siteSub" class="noprint">From Wikipedia, the free encyclopedia</div>				<div id="contentSub"></div>

<p><b>MapReduce</b> is a <a href="/wiki/Programming_model" name="Programming model">programming model</a> and an associated implementation for processing and generating <a href="/wiki/Big_data" name="Big data">big data</a> sets with a <a href="/wiki/Parallel_computing" name="Parallel computing">parallel</a>, <a href="/wiki/Distributed_computing" name="Distributed computing">distributed</a> algorithm on a <a href="/wiki/Cluster_(computing)" class="mw-redirect" name="Cluster (computing)">cluster</a>.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup><sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup>
</p><p>A MapReduce program is composed of a <a href="/wiki/Map_(parallel_pattern)" name="Map (parallel pattern)"><b>map</b></a> <a href="/wiki/Procedure_(computing)" class="mw-redirect" name="Procedure (computing)">procedure</a> (or method), which performs filtering and sorting (such as sorting students by first name into queues, one queue for each name), and a <i><a href="/wiki/Reduce_(parallel_pattern)" name="Reduce (parallel pattern)">reduce</a></i> method, which performs a summary operation (such as counting the number of students in each queue, yielding name frequencies). The "MapReduce System" (also called "infrastructure" or "framework") orchestrates the processing by <a href="/wiki/Marshalling_(computer_science)" name="Marshalling (computer science)">marshalling</a> the distributed servers, running the various tasks in parallel, managing all communications and data transfers between the various parts of the system, and providing for <a href="/wiki/Redundancy_(engineering)" name="Redundancy (engineering)">redundancy</a> and <a href="/wiki/Fault-tolerant_computer_system" name="Fault-tolerant computer system">fault tolerance</a>.
</p><p>The model is a specialization of the <i>split-apply-combine</i> strategy for data analysis.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup>
It is inspired by the <a href="/wiki/Map_(higher-order_function)" name="Map (higher-order function)">map</a> and <a href="/wiki/Fold_(higher-order_function)" name="Fold (higher-order function)">reduce</a> functions commonly used in <a href="/wiki/Functional_programming" name="Functional programming">functional programming</a>,<sup id="cite_ref-map_4-0" class="reference"><a href="#cite_note-map-4">[4]</a></sup> although their purpose in the MapReduce framework is not the same as in their original forms.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup> The key contributions of the MapReduce framework are not the actual map and reduce functions (which, for example, resemble the 1995 <a href="/wiki/Message_Passing_Interface" name="Message Passing Interface">Message Passing Interface</a> standard's<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup> <i>reduce</i><sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup> and <i>scatter</i><sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup> operations), but the scalability and fault-tolerance achieved for a variety of applications by optimizing the execution engine. As such, a <a href="/wiki/Single-threaded" class="mw-redirect" name="Single-threaded">single-threaded</a> implementation of MapReduce will usually not be faster than a traditional (non-MapReduce) implementation; any gains are usually only seen with <a href="/wiki/Multi-threaded" class="mw-redirect" name="Multi-threaded">multi-threaded</a> implementations on multi-processor hardware.<sup id="cite_ref-stackoverflow_9-0" class="reference"><a href="#cite_note-stackoverflow-9">[9]</a></sup> The use of this model is beneficial only when the optimized distributed shuffle operation (which reduces network communication cost) and fault tolerance features of the MapReduce framework come into play. Optimizing the communication cost is essential to a good MapReduce algorithm.<sup id="cite_ref-ullman_10-0" class="reference"><a href="#cite_note-ullman-10">[10]</a></sup>
</p><p>MapReduce <a href="/wiki/Library_(software)" class="mw-redirect" name="Library (software)">libraries</a> have been written in many programming languages, with different levels of optimization. A popular <a href="/wiki/Open-source_software" name="Open-source software">open-source</a> implementation that has support for distributed shuffles is part of <a href="/wiki/Apache_Hadoop" name="Apache Hadoop">Apache Hadoop</a>. The name MapReduce originally referred to the proprietary <a href="/wiki/Google" name="Google">Google</a> technology, but has since been <a href="/wiki/Generic_trademark" name="Generic trademark">genericized</a>. By 2014, Google was no longer using MapReduce as their primary <i><a href="/wiki/Big_data" name="Big data">big data</a></i> processing model,<sup id="cite_ref-11" class="reference"><a href="#cite_note-11">[11]</a></sup> and development on <a href="/wiki/Apache_Mahout" name="Apache Mahout">Apache Mahout</a> had moved on to more capable and less disk-oriented mechanisms that incorporated full map and reduce capabilities.<sup id="cite_ref-12" class="reference"><a href="#cite_note-12">[12]</a></sup>
</p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?name=MapReduce&amp;action=edit&amp;section=1" name="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>MapReduce is a framework for processing <a href="/wiki/Parallel_computing" name="Parallel computing">parallelizable</a> problems across large datasets using a large number of computers (nodes), collectively referred to as a <a href="/wiki/Computer_cluster" name="Computer cluster">cluster</a> (if all nodes are on the same local network and use similar hardware) or a <a href="/wiki/Grid_Computing" class="mw-redirect" name="Grid Computing">grid</a> (if the nodes are shared across geographically and administratively distributed systems, and use more heterogeneous hardware). Processing can occur on data stored either in a <a href="/wiki/Filesystem" class="mw-redirect" name="Filesystem">filesystem</a> (unstructured) or in a <a href="/wiki/Database" name="Database">database</a> (structured). MapReduce can take advantage of the locality of data, processing it near the place it is stored in order to minimize communication overhead.
</p><p>A MapReduce framework (or system) is usually composed of three operations (or steps):
</p>
<ol><li><b>Map:</b> each worker node applies the <code>map</code> function to the local data, and writes the output to a temporary storage. A master node ensures that only one copy of redundant input data is processed.</li>
<li><b>Shuffle:</b> worker nodes redistribute data based on the output keys (produced by the <code>map</code> function), such that all data belonging to one key is located on the same worker node.</li>
<li><b>Reduce:</b> worker nodes now process each group of output data, per key, in parallel.</li></ol>
<p>MapReduce allows for distributed processing of the map and reduction operations. Maps can be performed in parallel, provided that each mapping operation is independent of the others; in practice, this is limited by the number of independent data sources and/or the number of CPUs near each source. Similarly, a set of 'reducers' can perform the reduction phase, provided that all outputs of the map operation that share the same key are presented to the same reducer at the same time, or that the reduction function is <a href="/wiki/Associative_property" name="Associative property">associative</a>. While this process can often appear inefficient compared to algorithms that are more sequential (because multiple instances of the reduction process must be run), MapReduce can be applied to significantly larger datasets than a single <a href="/wiki/Commodity_computing" name="Commodity computing">"commodity" server</a> can handle&nbsp;– a large <a href="/wiki/Server_farm" name="Server farm">server farm</a> can use MapReduce to sort a <a href="/wiki/Petabyte" name="Petabyte">petabyte</a> of data in only a few hours.<sup id="cite_ref-13" class="reference"><a href="#cite_note-13">[13]</a></sup> The parallelism also offers some possibility of recovering from partial failure of servers or storage during the operation: if one mapper or reducer fails, the work can be rescheduled&nbsp;– assuming the input data is still available.
</p><p>Another way to look at MapReduce is as a 5-step parallel and distributed computation:
</p>
<ol><li><b>Prepare the Map() input</b> – the "MapReduce system" designates Map processors, assigns the input key value <i>K1</i> that each processor would work on, and provides that processor with all the input data associated with that key value.</li>
<li><b>Run the user-provided Map() code</b> – Map() is run exactly once for each <i>K1</i> key value, generating output organized by key values <i>K2</i>.</li>
<li><b>"Shuffle" the Map output to the Reduce processors</b> – the MapReduce system designates Reduce processors, assigns the <i>K2</i> key value each processor should work on, and provides that processor with all the Map-generated data associated with that key value.</li>
<li><b>Run the user-provided Reduce() code</b> – Reduce() is run exactly once for each <i>K2</i> key value produced by the Map step.</li>
<li><b>Produce the final output</b> – the MapReduce system collects all the Reduce output, and sorts it by <i>K2</i> to produce the final outcome.</li></ol>
<p>These five steps can be logically thought of as running in sequence – each step starts only after the previous step is completed – although in practice they can be interleaved as long as the final result is not affected.
</p><p>In many situations, the input data might already be distributed (<a href="/wiki/Shard_(database_architecture)" name="Shard (database architecture)">"sharded"</a>) among many different servers, in which case step 1 could sometimes be greatly simplified by assigning Map servers that would process the locally present input data.  Similarly, step 3 could sometimes be sped up by assigning Reduce processors that are as close as possible to the Map-generated data they need to process.
</p>
<h2><span class="mw-headline" id="Logical_view">Logical view</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?name=MapReduce&amp;action=edit&amp;section=2" name="Edit section: Logical view">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The  <i>Map</i> and <i>Reduce</i> functions of <i>MapReduce</i> are both defined with respect to data structured in (key, value) pairs. <i>Map</i> takes one pair of data with a type in one <a href="/wiki/Data_domain" name="Data domain">data domain</a>, and returns a list of pairs in a different domain:
</p><p><code>Map(k1,v1)</code> → <code>list(k2,v2)</code>
</p><p>The <i>Map</i> function is applied in parallel to every pair (keyed by <code>k1</code>) in the input dataset. This produces a list of pairs (keyed by <code>k2</code>) for each call.
After that, the MapReduce framework collects all pairs with the same key (<code>k2</code>) from all lists and groups them together, creating one group for each key.
</p><p>The <i>Reduce</i> function is then applied in parallel to each group, which in turn produces a collection of values in the same domain:
</p><p><code>Reduce(k2, list (v2))</code> → <code>list(v3)</code>
</p><p>Each <i>Reduce</i> call typically produces either one value v3 or an empty return, though one call is allowed to return more than one value. The returns of all calls are collected as the desired result list.
</p><p>Thus the MapReduce framework transforms a list of (key, value) pairs into a list of values. This behavior is different from the typical functional programming map and reduce combination, which accepts a list of arbitrary values and returns one single value that combines <i>all</i> the values returned by map.
</p><p>It is <a href="/wiki/Necessity_and_sufficiency" name="Necessity and sufficiency">necessary but not sufficient</a> to have implementations of the map and reduce abstractions in order to implement MapReduce. Distributed implementations of MapReduce require a means of connecting the processes performing the Map and Reduce phases. This may be a <a href="/wiki/Distributed_file_system" class="mw-redirect" name="Distributed file system">distributed file system</a>. Other options are possible, such as direct streaming from mappers to reducers, or for the mapping processors to serve up their results to reducers that query them.
</p>
<h3><span class="mw-headline" id="Examples">Examples</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?name=MapReduce&amp;action=edit&amp;section=3" name="Edit section: Examples">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The canonical MapReduce example counts the appearance of each word in a set of documents:<sup id="cite_ref-14" class="reference"><a href="#cite_note-14">[14]</a></sup>
</p>
<pre><b>function</b> <u>map</u>(String name, String document):
  <i>// name: document name</i>
  <i>// document: document contents</i>
  <b>for each</b> word w <b>in</b> document:
    emit (w, 1)

<b>function</b> <u>reduce</u>(String word, Iterator partialCounts):
  <i>// word: a word</i>
  <i>// partialCounts: a list of aggregated partial counts</i>
  sum = 0
  <b>for each</b> pc <b>in</b> partialCounts:
    sum += pc
  emit (word, sum)
</pre>
<p>Here, each document is split into words, and each word is counted by the <i>map</i> function, using the word as the result key. The framework puts together all the pairs with the same key and feeds them to the same call to <i>reduce</i>. Thus, this function just needs to sum all of its input values to find the total appearances of that word.
</p><p>As another example, imagine that for a database of 1.1 billion people, one would like to compute the average number of social contacts a person has according to age.  In <a href="/wiki/SQL" name="SQL">SQL</a>, such a query could be expressed as:
</p>
<div class="mw-highlight mw-content-ltr" dir="ltr"><pre><span></span>  <span class="k">SELECT</span> <span class="n">age</span><span class="p">,</span> <span class="k">AVG</span><span class="p">(</span><span class="n">contacts</span><span class="p">)</span>
    <span class="k">FROM</span> <span class="n">social</span><span class="p">.</span><span class="n">person</span>
<span class="k">GROUP</span> <span class="k">BY</span> <span class="n">age</span>
<span class="k">ORDER</span> <span class="k">BY</span> <span class="n">age</span>
</pre></div>
<p>Using MapReduce, the <tt>K1</tt> key values could be the integers 1 through 1100, each representing a batch of 1 million records, the <tt>K2</tt> key value could be a person's age in years, and this computation could be achieved using the following functions:
</p>
<pre><b>function</b> Map <b>is</b>
    <b>input:</b> <b>integer</b> K1 between 1 and 1100, representing a batch of 1 million social.person records
    <b>for each</b> social.person record in the K1 batch <b>do</b>
        <b>let</b> Y be the person's age
        <b>let</b> N be the number of contacts the person has
        <b>produce one output record</b> (Y,(N,1))
    <b>repeat</b>
<b>end function</b>

<b>function</b> Reduce <b>is</b>
    <b>input:</b> age (in years) Y
    <b>for each</b> input record (Y,(N,C)) <b>do</b>
        Accumulate in S the sum of N*C
        Accumulate in C<sub>new</sub> the sum of C
    <b>repeat</b>
    <b>let</b> A be S/C<sub>new</sub>
    <b>produce one output record</b> (Y,(A,C<sub>new</sub>))
<b>end function</b>
</pre>
<p>The MapReduce system would line up the 1100 Map processors, and would provide each with its corresponding 1 million input records. The Map step would produce 1.1 billion <tt>(Y,(N,1))</tt> records, with <tt>Y</tt> values ranging between, say, 8 and 103. The MapReduce System would then line up the 96 Reduce processors by performing shuffling operation of the key/value pairs due to the fact that we need average per age, and provide each with its millions of corresponding input records. The Reduce step would result in the much reduced set of only 96 output records <tt>(Y,A)</tt>, which would be put in the final result file, sorted by <tt>Y</tt>.
</p><p>The count info in the record is important if the processing is reduced more than one time. If we did not add the count of the records, the computed average would be wrong, for example:
</p>
<pre><i>-- map output #1: age, quantity of contacts</i>
10, 9
10, 9
10, 9
</pre>
<pre><i>-- map output #2: age, quantity of contacts</i>
10, 9
10, 9
</pre>
<pre><i>-- map output #3: age, quantity of contacts</i>
10, 10
</pre>
<p>If we reduce files <tt>#1</tt> and <tt>#2</tt>, we will have a new file with an average of 9 contacts for a 10-year-old person ((9+9+9+9+9)/5):
</p>
<pre><i>-- reduce step #1: age, average of contacts</i>
10, 9
</pre>
<p>If we reduce it with file <tt>#3</tt>, we lose the count of how many records we've already seen, so we end up with an average of 9.5 contacts for a 10-year-old person ((9+10)/2), which is wrong. The correct answer is 9.1<span style="text-decoration: overline;">66</span> = 55 / 6 = (9*3+9*2+10*1)/(3+2+1).
</p>
<h2><span class="mw-headline" id="Dataflow">Dataflow</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?name=MapReduce&amp;action=edit&amp;section=4" name="Edit section: Dataflow">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The frozen part<sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Please_clarify" name="Wikipedia:Please clarify"><span name="What exactly is meant here by &quot;frozen part&quot;? (March 2018)">clarification needed</span></a></i>]</sup> of the MapReduce framework is a large distributed sort. The hot spots, which the application defines, are:
</p>
<ul><li>an <i>input reader</i></li>
<li>a <i>Map</i> function</li>
<li>a <i>partition</i> function</li>
<li>a <i>compare</i> function</li>
<li>a <i>Reduce</i> function</li>
<li>an <i>output writer</i></li></ul>
<h3><span class="mw-headline" id="Input_reader">Input reader</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?name=MapReduce&amp;action=edit&amp;section=5" name="Edit section: Input reader">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <i>input reader</i> divides the input into appropriate size 'splits' (in practice, typically, 64&nbsp;MB to 128&nbsp;MB) and the framework assigns one split to each <i>Map</i> function. The <i>input reader</i> reads data from stable storage (typically, a <a href="/wiki/Distributed_file_system" class="mw-redirect" name="Distributed file system">distributed file system</a>) and generates key/value pairs.
</p><p>A common example will read a directory full of text files and return each line as a record.
</p>
<h3><span class="mw-headline" id="Map_function">Map function</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?name=MapReduce&amp;action=edit&amp;section=6" name="Edit section: Map function">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <i>Map</i> function takes a series of key/value pairs, processes each, and generates zero or more output key/value pairs. The input and output types of the map can be (and often are) different from each other.
</p><p>If the application is doing a word count, the map function would break the line into words and output a key/value pair for each word.  Each output pair would contain the word as the key and the number of instances of that word in the line as the value.
</p>
<h3><span class="mw-headline" id="Partition_function">Partition function</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?name=MapReduce&amp;action=edit&amp;section=7" name="Edit section: Partition function">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Each <i>Map</i> function output is allocated to a particular <i>reducer</i> by the application's <i>partition</i> function for <a href="/wiki/Sharding" class="mw-redirect" name="Sharding">sharding</a> purposes. The <i>partition</i> function is given the key and the number of reducers and returns the index of the desired <i>reducer</i>.
</p><p>A typical default is to <a href="/wiki/Hash_function" name="Hash function">hash</a> the key and use the hash value <a href="/wiki/Modulo_operation" name="Modulo operation">modulo</a> the number of <i>reducers</i>. It is important to pick a partition function that gives an approximately uniform distribution of data per shard for <a href="/wiki/Load_balancing_(computing)" name="Load balancing (computing)">load-balancing</a> purposes, otherwise the MapReduce operation can be held up waiting for slow reducers to finish
(i.e. the reducers assigned the larger shares of the non-uniformly partitioned data).
</p><p>Between the map and reduce stages, the data are <i>shuffled</i> (parallel-sorted / exchanged between nodes) in order to move the data from the map node that produced them to the shard in which they will be reduced.  The shuffle can sometimes take longer than the computation time depending on network bandwidth, CPU speeds, data produced and time taken by map and reduce computations.
</p>
<h3><span class="mw-headline" id="Comparison_function">Comparison function</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?name=MapReduce&amp;action=edit&amp;section=8" name="Edit section: Comparison function">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The input for each <i>Reduce</i> is pulled from the machine where the <i>Map</i> ran and sorted using the application's <i>comparison</i> function.
</p>
<h3><span class="mw-headline" id="Reduce_function">Reduce function</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?name=MapReduce&amp;action=edit&amp;section=9" name="Edit section: Reduce function">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The framework calls the application's <i>Reduce</i> function once for each unique key in the sorted order. The <i>Reduce</i> can iterate through the values that are associated with that key and produce zero or more outputs.
</p><p>In the word count example, the <i>Reduce</i> function takes the input values, sums them and generates a single output of the word and the final sum.
</p>
<h3><span class="mw-headline" id="Output_writer">Output writer</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?name=MapReduce&amp;action=edit&amp;section=10" name="Edit section: Output writer">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <i>Output Writer</i> writes the output of the <i>Reduce</i> to the stable storage.
</p>
<h2><span class="mw-headline" id="Performance_considerations">Performance considerations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?name=MapReduce&amp;action=edit&amp;section=11" name="Edit section: Performance considerations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>MapReduce programs are not guaranteed to be fast. The main benefit of this programming model is to exploit the optimized shuffle operation of the platform, and only having to write the <i>Map</i> and <i>Reduce</i> parts of the program.
In practice, the author of a MapReduce program however has to take the shuffle step into consideration; in particular the partition function and the amount of data written by the <i>Map</i> function can have a large impact on the performance and scalability. Additional modules such as the <i>Combiner</i> function can help to reduce the amount of data written to disk, and transmitted over the network. MapReduce applications can achieve sub-linear speedups under specific circumstances.<sup id="cite_ref-:0_15-0" class="reference"><a href="#cite_note-:0-15">[15]</a></sup>
</p><p>When designing a MapReduce algorithm, the author needs to choose a good tradeoff<sup id="cite_ref-ullman_10-1" class="reference"><a href="#cite_note-ullman-10">[10]</a></sup> between the computation and the communication costs. Communication cost often dominates the computation cost,<sup id="cite_ref-ullman_10-2" class="reference"><a href="#cite_note-ullman-10">[10]</a></sup><sup id="cite_ref-:0_15-1" class="reference"><a href="#cite_note-:0-15">[15]</a></sup> and many MapReduce implementations are designed to write all communication to distributed storage for crash recovery.
</p><p>In tuning performance of MapReduce, the complexity of mapping, shuffle, sorting (grouping by the key), and reducing has to be taken into account. The amount of data produced by the mappers is a key parameter that shifts the bulk of the computation cost between mapping and reducing. Reducing includes sorting (grouping of the keys) which has nonlinear complexity. Hence, small partition sizes reduce sorting time, but there is a trade-off because having a large number of reducers may be impractical. The influence of split unit size is marginal (unless chosen particularly badly, say &lt;1MB). The gains from some mappers reading load from local disks, on average, is minor.<sup id="cite_ref-16" class="reference"><a href="#cite_note-16">[16]</a></sup>
</p><p>For processes that complete quickly, and where the data fits into main memory of a single machine or a small cluster, using a MapReduce framework usually is not effective. Since these frameworks are designed to recover from the loss of whole nodes during the computation, they write interim results to distributed storage. This crash recovery is expensive, and only pays off when the computation involves many computers and a long runtime of the computation. A task that completes in seconds can just be restarted in the case of an error, and the likelihood of at least one machine failing grows quickly with the cluster size. On such problems, implementations keeping all data in memory and simply restarting a computation on node failures or —when the data is small enough— non-distributed solutions will often be faster than a MapReduce system.
</p>
<h2><span class="mw-headline" id="Distribution_and_reliability">Distribution and reliability</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?name=MapReduce&amp;action=edit&amp;section=12" name="Edit section: Distribution and reliability">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>MapReduce achieves reliability by parceling out a number of operations on the set of data to each node in the network. Each node is expected to report back periodically with completed work and status updates. If a node falls silent for longer than that interval, the master node (similar to the master server in the <a href="/wiki/Google_File_System" name="Google File System">Google File System</a>) records the node as dead and sends out the node's assigned work to other nodes. Individual operations use <a href="/wiki/Atomicity_(programming)" class="mw-redirect" name="Atomicity (programming)">atomic</a> operations for naming file outputs as a check to ensure that there are not parallel conflicting threads running. When files are renamed, it is possible to also copy them to another name in addition to the name of the task (allowing for <a href="/wiki/Side-effect_(computer_science)" class="mw-redirect" name="Side-effect (computer science)">side-effects</a>).
</p><p>The reduce operations operate much the same way. Because of their inferior properties with regard to parallel operations, the master node attempts to schedule reduce operations on the same node, or in the same rack as the node holding the data being operated on. This property is desirable as it conserves bandwidth across the backbone network of the datacenter.
</p><p>Implementations are not necessarily highly reliable. For example, in older versions of <a href="/wiki/Hadoop" class="mw-redirect" name="Hadoop">Hadoop</a> the <i>NameNode</i> was a <a href="/wiki/Single_point_of_failure" name="Single point of failure">single point of failure</a> for the distributed filesystem.  Later versions of Hadoop have high availability with an active/passive failover for the "NameNode."
</p>
<h2><span class="mw-headline" id="Uses">Uses</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?name=MapReduce&amp;action=edit&amp;section=13" name="Edit section: Uses">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>MapReduce is useful in a wide range of applications, including distributed pattern-based searching, distributed sorting, web link-graph reversal, Singular Value Decomposition,<sup id="cite_ref-17" class="reference"><a href="#cite_note-17">[17]</a></sup> web access log stats, <a href="/wiki/Inverted_index" name="Inverted index">inverted index</a> construction, <a href="/wiki/Document_clustering" name="Document clustering">document clustering</a>, <a href="/wiki/Machine_learning" name="Machine learning">machine learning</a>,<sup id="cite_ref-mrml_18-0" class="reference"><a href="#cite_note-mrml-18">[18]</a></sup> and <a href="/wiki/Statistical_machine_translation" name="Statistical machine translation">statistical machine translation</a>. Moreover, the MapReduce model has been adapted to several computing environments like multi-core and many-core systems,<sup id="cite_ref-evalMR_19-0" class="reference"><a href="#cite_note-evalMR-19">[19]</a></sup><sup id="cite_ref-graphicsMR_20-0" class="reference"><a href="#cite_note-graphicsMR-20">[20]</a></sup><sup id="cite_ref-tiledMR_21-0" class="reference"><a href="#cite_note-tiledMR-21">[21]</a></sup> desktop grids,<sup id="cite_ref-gridMR_22-0" class="reference"><a href="#cite_note-gridMR-22">[22]</a></sup>
multi-cluster,<sup id="cite_ref-HMR_23-0" class="reference"><a href="#cite_note-HMR-23">[23]</a></sup> volunteer computing environments,<sup id="cite_ref-volunteerMR_24-0" class="reference"><a href="#cite_note-volunteerMR-24">[24]</a></sup> dynamic cloud environments,<sup id="cite_ref-dynCloudMR_25-0" class="reference"><a href="#cite_note-dynCloudMR-25">[25]</a></sup> mobile environments,<sup id="cite_ref-mobileMR_26-0" class="reference"><a href="#cite_note-mobileMR-26">[26]</a></sup> and high-performance computing environments.<sup id="cite_ref-27" class="reference"><a href="#cite_note-27">[27]</a></sup>
</p><p>At Google, MapReduce was used to completely regenerate Google's index of the <a href="/wiki/World_Wide_Web" name="World Wide Web">World Wide Web</a>. It replaced the old <i>ad hoc</i> programs that updated the index and ran the various analyses.<sup id="cite_ref-usage_28-0" class="reference"><a href="#cite_note-usage-28">[28]</a></sup> Development at Google has since moved on to technologies such as Percolator, FlumeJava<sup id="cite_ref-Chambers2010_29-0" class="reference"><a href="#cite_note-Chambers2010-29">[29]</a></sup> and <a href="/wiki/Google_MillWheel" name="Google MillWheel">MillWheel</a> that offer streaming operation and updates instead of batch processing, to allow integrating "live" search results without rebuilding the complete index.<sup id="cite_ref-30" class="reference"><a href="#cite_note-30">[30]</a></sup>
</p><p>MapReduce's stable inputs and outputs are usually stored in a <a href="/wiki/Distributed_file_system" class="mw-redirect" name="Distributed file system">distributed file system</a>. The transient data are usually stored on local disk and fetched remotely by the reducers.
</p>
<h2><span class="mw-headline" id="Criticism">Criticism</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?name=MapReduce&amp;action=edit&amp;section=14" name="Edit section: Criticism">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Lack_of_novelty">Lack of novelty</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?name=MapReduce&amp;action=edit&amp;section=15" name="Edit section: Lack of novelty">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/David_DeWitt" name="David DeWitt">David DeWitt</a> and <a href="/wiki/Michael_Stonebraker" name="Michael Stonebraker">Michael Stonebraker</a>, computer scientists specializing in <a href="/wiki/Parallel_database" name="Parallel database">parallel databases</a> and <a href="/wiki/Shared-nothing_architecture" name="Shared-nothing architecture">shared-nothing architectures</a>, have been critical of the breadth of problems that MapReduce can be used for.<sup id="cite_ref-shark_31-0" class="reference"><a href="#cite_note-shark-31">[31]</a></sup> They called its interface too low-level and questioned whether it really represents the <a href="/wiki/Paradigm_shift" name="Paradigm shift">paradigm shift</a> its proponents have claimed it is.<sup id="cite_ref-ddandms1_32-0" class="reference"><a href="#cite_note-ddandms1-32">[32]</a></sup> They challenged the MapReduce proponents' claims of novelty, citing <a href="/wiki/Teradata" name="Teradata">Teradata</a> as an example of <a href="/wiki/Prior_art" name="Prior art">prior art</a> that has existed for over two decades. They also compared MapReduce programmers to <a href="/wiki/CODASYL" name="CODASYL">CODASYL</a> programmers, noting both are "writing in a <a href="/wiki/Low-level_programming_language" name="Low-level programming language">low-level language</a> performing low-level record manipulation."<sup id="cite_ref-ddandms1_32-1" class="reference"><a href="#cite_note-ddandms1-32">[32]</a></sup> MapReduce's use of input files and lack of <a href="/wiki/Logical_schema" class="mw-redirect" name="Logical schema">schema</a> support prevents the performance improvements enabled by common database system features such as <a href="/wiki/B-tree" name="B-tree">B-trees</a> and <a href="/wiki/Partition_(database)" name="Partition (database)">hash partitioning</a>, though projects such as <a href="/wiki/Pig_(programming_language)" class="mw-redirect" name="Pig (programming language)">Pig (or PigLatin)</a>, <a href="/wiki/Sawzall_(programming_language)" name="Sawzall (programming language)">Sawzall</a>, <a href="/wiki/Apache_Hive" name="Apache Hive">Apache Hive</a>,<sup id="cite_ref-ApacheHiveWiki_33-0" class="reference"><a href="#cite_note-ApacheHiveWiki-33">[33]</a></sup> <a rel="nofollow" class="external text" href="https://archive.is/20121214201610/http://ysmart.cse.ohio-state.edu/">YSmart</a>,<sup id="cite_ref-YSmartPaper_34-0" class="reference"><a href="#cite_note-YSmartPaper-34">[34]</a></sup> <a href="/wiki/HBase" class="mw-redirect" name="HBase">HBase</a><sup id="cite_ref-HBase_35-0" class="reference"><a href="#cite_note-HBase-35">[35]</a></sup> and <a href="/wiki/Bigtable" name="Bigtable">Bigtable</a><sup id="cite_ref-HBase_35-1" class="reference"><a href="#cite_note-HBase-35">[35]</a></sup><sup id="cite_ref-BigtablePaper_36-0" class="reference"><a href="#cite_note-BigtablePaper-36">[36]</a></sup> are addressing some of these problems.
</p><p>Greg Jorgensen wrote an article rejecting these views.<sup id="cite_ref-gj1_37-0" class="reference"><a href="#cite_note-gj1-37">[37]</a></sup> Jorgensen asserts that DeWitt and Stonebraker's entire analysis is groundless as MapReduce was never designed nor intended to be used as a database.
</p><p>DeWitt and Stonebraker have subsequently published a detailed benchmark study in 2009 comparing performance of <a href="/wiki/Hadoop" class="mw-redirect" name="Hadoop">Hadoop's</a> MapReduce and <a href="/wiki/RDBMS" class="mw-redirect" name="RDBMS">RDBMS</a> approaches on several specific problems.<sup id="cite_ref-sigmod_38-0" class="reference"><a href="#cite_note-sigmod-38">[38]</a></sup> They concluded that relational databases offer real advantages for many kinds of data use, especially on complex processing or where the data is used across an enterprise, but that MapReduce may be easier for users to adopt for simple or one-time processing tasks.
</p><p>Google has been granted a patent on MapReduce.<sup id="cite_ref-patent_39-0" class="reference"><a href="#cite_note-patent-39">[39]</a></sup> However, there have been claims that this patent should not have been granted because MapReduce is too similar to existing products. For example, map and reduce functionality can be very easily implemented in <a href="/wiki/Oracle_database" class="mw-redirect" name="Oracle database">Oracle's</a> <a href="/wiki/PL/SQL" name="PL/SQL">PL/SQL</a> database oriented language<sup id="cite_ref-Curt_Monash_40-0" class="reference"><a href="#cite_note-Curt_Monash-40">[40]</a></sup> or is supported for developers transparently in distributed database architectures such as <a href="/wiki/Clusterpoint" name="Clusterpoint">Clusterpoint</a> XML database<sup id="cite_ref-Clusterpoint_41-0" class="reference"><a href="#cite_note-Clusterpoint-41">[41]</a></sup> or <a href="/wiki/MongoDB" name="MongoDB">MongoDB</a> NoSQL database.<sup id="cite_ref-MongoDB_42-0" class="reference"><a href="#cite_note-MongoDB-42">[42]</a></sup>
</p>
<h3><span class="mw-headline" id="Restricted_programming_framework">Restricted programming framework</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?name=MapReduce&amp;action=edit&amp;section=16" name="Edit section: Restricted programming framework">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>MapReduce tasks must be written as acyclic dataflow programs, i.e. a stateless mapper followed by a stateless reducer, that are executed by a batch job scheduler. This paradigm makes repeated querying of datasets difficult and imposes limitations that are felt in fields such as <a href="/wiki/Machine_learning" name="Machine learning">machine learning</a>, where iterative algorithms that revisit a single <a href="/wiki/Working_set" name="Working set">working set</a> multiple times are the norm.<sup id="cite_ref-43" class="reference"><a href="#cite_note-43">[43]</a></sup>
</p>
